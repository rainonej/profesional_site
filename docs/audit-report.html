About this Report

This audit covers the complete professional_site repository as of the main branch. The site is a personal academic / portfolio website for Agreni Batra built with Astro 4, Tailwind CSS 3, and Pages CMS for content management. Deployment is via GitHub Actions to GitHub Pages at rainonej.github.io/profesional_site/.

The primary content editor interface (Pages CMS) runs at app.pagescms.org. A secondary Decap CMS admin panel lives at /admin/ and requires Netlify Identity — which is not connected.

Bugs / Errors
Things breaking right now
Pages CMS: media property is at the wrong level on image fields
Bug

In .pages.yml lines 35, 59, image fields use media: media as a top-level field property. According to the Pages CMS documentation, media must be nested inside an options: block. Using it at the top level causes the "Property 'media' isn't valid and will be ignored" warning in the CMS UI, meaning the image fields are not linked to the correct media folder. Editors uploading a profile photo or project image via the Image picker may not see files from site/public/media/.

Affected fields: photo (settings) at line 34–35 and image (projects) at line 58–59.

# CURRENT (broken) — top-level media property
      - name: photo
        label: Photo
        type: image
        media: media     # invalid at this level

# CORRECT — media nested inside options
      - name: photo
        label: Photo
        type: image
        options:
          media: media   # valid

See the Pages CMS Fix section below for the full corrected .pages.yml snippet.

Fix
Move media: media inside an options: key on both image fields in .pages.yml. See the dedicated Pages CMS Fix section for the complete corrected file.

Warnings
Things that will cause problems soon
Netlify Identity widget loaded on every page — site is on GitHub Pages
Warning

In site/src/layouts/Layout.astro line 28, the Netlify Identity widget script (netlify-identity-widget.js) is loaded on every page of the site:

<script src="https://identity.netlify.com/v1/netlify-identity-widget.js"></script>

Netlify Identity only works when the site is hosted on Netlify with the Identity feature enabled. This site is deployed to GitHub Pages, where Netlify Identity has no backend to connect to. Every visitor therefore downloads ~50 KB of JavaScript that does nothing. The accompanying inline script block at lines 40–49 also fires and looks for window.netlifyIdentity, which may throw errors in strict environments.

The widget was added to support Decap CMS authentication, but Decap CMS also cannot be used on GitHub Pages without a Netlify backend. The Decap CMS files in site/public/admin/ are effectively non-functional in the current deployment.

Fix
Remove the Netlify Identity <script> tag from Layout.astro (line 28) and the inline netlifyIdentity handler (lines 39–50). If Decap CMS is desired in future, migrate to Netlify hosting or use a Git-based backend with OAuth (e.g., Decap's local_backend: true for development, combined with a serverless token endpoint for production).
Decap CMS config.yml points to wrong file paths
Warning

In site/public/admin/config.yml line 9–10, Decap CMS is configured with:

media_folder: public/images/uploads
public_folder: /images/uploads

However, the actual media folder used by Pages CMS (and the place the site expects images) is site/public/media/, not public/images/uploads. There is also a separate site/public/images/uploads/.gitkeep directory — meaning two different upload paths exist, neither is consistently used.

Additionally, Decap's file: path for settings points to src/content/settings/main.json (line 22) without the site/ prefix — this is relative to where Decap CMS writes files, which depends on how decap-server is run. This can cause Decap to write to the wrong location.

Fix
If Decap CMS is kept, unify the media path to match Pages CMS: set media_folder: public/media and public_folder: /media in site/public/admin/config.yml. Delete site/public/images/uploads/. If Decap is dropped in favour of Pages CMS only, delete the entire site/public/admin/ directory.
CI workflow uploads artifact from wrong path on non-main branches
Warning

In .github/workflows/ci.yml line 36, the artifact upload step has:

path: site/dist

The defaults.run.working-directory is set to site (line 15), which means shell commands like npm run build run from inside the site/ directory. However, upload-pages-artifact is not a shell command — its path parameter is relative to the workspace root, not the working directory. So site/dist is the correct path here.

This is currently working, but it is a subtle inconsistency that could break if the working directory default is removed. More importantly, there is no caching of node_modules across runs, so every CI run does a full npm install (potentially 60–90s of unnecessary wait time).

Fix
Add a cache: 'npm' option to the actions/setup-node@v4 step to cache node_modules between runs. This reduces CI time from ~90s to ~15s on cache hit. Also add a npm ci instead of npm install for reproducible installs (requires a package-lock.json).
No package-lock.json — CI installs may produce different package versions over time
Warning

The site/package.json file uses range specifiers (^4.16.18, ^3.4.14, ^5.1.0) but there is no package-lock.json committed to the repository. CI runs npm install, which resolves the latest compatible versions at install time. This means today's build may use different package versions than tomorrow's build if a minor or patch release is published.

The CI workflow at line 28 runs npm install rather than npm ci. npm ci is the correct command for CI environments: it requires a lockfile, installs exactly the locked versions, and fails fast if the lockfile is out of sync.

Fix
Run npm install locally once, commit the generated site/package-lock.json, then change the CI step to npm ci. Add package-lock.json to the set of files that trigger CI invalidation.

No content.config.ts — type-safe content collections not set up
Missing

Astro's content collections require a site/src/content.config.ts (or .js / .mjs) file that defines the schema for each collection using Zod. This project uses Astro.glob() to load Markdown files instead, which provides no type checking, no frontmatter validation, and no build-time error if a required field is missing or has the wrong type.

For example, if a project's Markdown file omits the date field, the sort at site/src/pages/work.astro lines 6–8 silently fails to sort correctly because new Date(undefined) returns NaN.

Fix
Create site/src/content.config.ts, define projects and settings collections with Zod schemas, and replace Astro.glob() calls with getCollection('projects'). This gives build-time type errors for missing or wrong-type frontmatter.
No individual project detail pages
Missing

The body content in Markdown files is never displayed — the work grid only shows title, description, tags, and an optional external link.

For a portfolio site, being able to click into a project and read a full case study is essential. For a PhD application site, detailed writeups of curriculum projects are the primary evidence of research and practice.

Create site/src/pages/work/[slug].astro as a dynamic route using getStaticPaths() with content collections. Render the project body with <Content /> from Astro's Markdown rendering. Link the work grid cards to their detail pages.

Best Practice Violations
Things that work but shouldn't be done this way
Astro.glob() used instead of Content Collections API
Best Practice

Both site/src/pages/index.astro line 6 and site/src/pages/work.astro line 5 use Astro.glob() to load project Markdown files:

const projects = await Astro.glob('../content/projects/*.md');

Astro.glob() is the legacy pattern from Astro 1.x. It returns untyped objects with a frontmatter property that has type Record<string, any> — no validation, no autocomplete, no build-time errors for missing required fields.

Astro's Content Collections API (getCollection()) has been the recommended approach since Astro 2.0 and provides full TypeScript type safety derived from a Zod schema. In Astro 5.x, the Content Layer API extends this further. The project's tsconfig.json uses "extends": "astro/tsconfigs/strict", making the lack of type-safe content loading particularly inconsistent.

Fix
Create site/src/content.config.ts and replace Astro.glob() with getCollection('projects'). This is a pre-requisite for adding the writing/research section cleanly and for getting to individual project pages.
@astrojs/tailwind integration is deprecated
Best Practice

site/package.json line 12 and site/astro.config.mjs line 2 use @astrojs/tailwind with Tailwind CSS 3. This integration is deprecated according to the Astro documentation: "Tailwind CSS now offers a Vite plugin which is the preferred way to use Tailwind 4 in Astro."

The current setup (Tailwind v3 + @astrojs/tailwind) continues to work and is not broken, but it is on an end-of-life path. Tailwind CSS v4 released in 2025 uses an entirely different configuration model — no tailwind.config.mjs, no PostCSS, CSS-first configuration via @import "tailwindcss".

The current site/src/styles/global.css correctly uses the v3 three-directive syntax (@tailwind base; @tailwind components; @tailwind utilities;) which is valid and correct for Tailwind v3.

Fix
No urgent action required — Tailwind v3 + @astrojs/tailwind still builds correctly. When upgrading to Tailwind v4, remove @astrojs/tailwind, install @tailwindcss/vite, migrate tailwind.config.mjs settings into global.css, and change global.css to @import "tailwindcss". Defer this migration to a dedicated upgrade PR.
Calendly assets loaded inside <section> — invalid HTML
Best Practice

In site/src/pages/index.astro lines 44–45 and site/src/pages/contact.astro lines 19, 25, Calendly's <link> and <script> tags are placed inside a <section> element in the body, not in the <head>:

<!-- inside <section> in body -->
<link href="https://assets.calendly.com/assets/external/widget.css" rel="stylesheet" />
<script type="text/javascript" src="..." async></script>

<link rel="stylesheet"> tags in the body are technically invalid HTML (they belong in <head>). While browsers handle this tolerantly, it means the stylesheet loads after the page has started rendering, causing a flash of unstyled Calendly widget. The Astro Layout.astro layout has a <head> that accepts a slot — Calendly assets should be injected there via Astro's <Fragment slot="head"> pattern or a prop.

Fix
Move Calendly <link> and <script> tags to the Layout's <head> section, or add a head slot to Layout.astro so individual pages can inject head assets. Loading Calendly only on the contact page (and the index page popup) prevents loading it sitewide.
Settings loaded via direct JSON import — bypasses CMS schema validation
Best Practice

All four page files import site settings with:

import settings from '../content/settings/main.json';

This is direct JSON import with TypeScript inferring the type from the current JSON contents. If main.json is later edited to remove a field (e.g., email is deleted), pages that reference settings.email will have a TypeScript error that is hard to trace. More importantly, there is no declared schema for what the settings file should contain — the schema lives in .pages.yml and site/public/admin/config.yml but not in the Astro build.

Using a content collection with a type: 'data' loader and a Zod schema for the settings file would make this type-safe and schema-validated at build time.

Fix
In content.config.ts, define a settings collection using Astro's file() loader targeting main.json with a Zod schema. Use getEntry('settings', 'main') in pages instead of the raw JSON import.

Upgrade to Astro 5 + Tailwind CSS v4
Refactor

The current stack uses Astro 4.16 and Tailwind 3.4 — both are behind their current major versions. Astro 5 introduced the Content Layer API (a more powerful successor to v2 content collections), improved performance, and first-class Tailwind v4 support via npx astro add tailwind.

Tailwind v4 is a significant API change: no tailwind.config.mjs, CSS-first configuration, new @import "tailwindcss" directive. All custom theme values (font families in tailwind.config.mjs) would need migrating to CSS custom properties in global.css.

Recommended approach: upgrade Astro first (lower risk), then Tailwind in a separate PR.

What Agreni Can Do
Preview options, request form options, and self-service actions

Can I preview my changes before they go live?

Short answer: not natively in Pages CMS. Pages CMS does not currently offer a live preview panel. When you save in Pages CMS, it commits to GitHub, which triggers CI, which takes ~60 seconds to deploy. You are essentially previewing on the live site after a 60-second delay.

Option A — Run the site locally (developer required once)
A developer installs Node.js and runs npm run dev inside site/. You get a local version at localhost:4321 that live-reloads as you edit files. This is the best preview experience but requires a developer to set up your machine once.

Option B — Deploy a preview environment on Netlify (free tier)
Connect the repository to a free Netlify account. Netlify will build a separate preview URL for every branch and pull request. Pages CMS edits go to a draft branch, and Netlify builds a preview at draft--your-site.netlify.app before the changes hit main. This is the most practical no-code preview workflow and costs $0 on Netlify's free tier.

Option C — CloudCannon (paid)
CloudCannon is a visual CMS with a built-in live preview editor. It integrates with Astro natively and shows changes as you type. It costs ~$25–45/month — appropriate if visual editing becomes a regular workflow, not for a demo.

Can visitors contact me without email — as a "project request" form?

Yes, several options work well for a static site on GitHub Pages.

Option A — Formspree (recommended, free tier)
Formspree is the simplest option. You add a plain HTML form with action="https://formspree.io/f/YOUR_ID". Submissions are emailed to you and optionally stored in the Formspree dashboard. Free tier allows 50 submissions/month. No backend, no API key in the frontend, no CORS issues.

<form action="https://formspree.io/f/YOUR_FORM_ID" method="POST">
  <input type="email" name="email" placeholder="Your email" />
  <textarea name="message" placeholder="Describe your project"></textarea>
  <button type="submit">Send</button>
</form>

Option B — GitHub Issues as a request inbox
A contact form can POST to the GitHub REST API to create a new issue in a private repository. The submission appears in your GitHub Issues tab with all form fields visible. This requires exposing a personal access token or routing through a small serverless function (Vercel / Netlify function, free tier).

  • A form handler serverless function (50 lines of Node.js) receives the POST and calls POST /repos/{owner}/{repo}/issues
  • Deploy the function to Vercel (free) or Netlify Functions (free)
  • The GitHub repo receiving issues can be private (only you see submissions)
  • You can tag, assign, and track project requests like a lightweight CRM

This is more setup but very elegant for a "project brief" intake form.

Option C — Calendly (already integrated)
The existing Calendly embed on the Contact page effectively serves as the contact mechanism once the PLACEHOLDER username is replaced. Visitors book 30-minute conversations directly without needing a form. This is often the best UX for consulting intake.
