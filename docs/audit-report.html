Code Review Delta (2026-04-01)

Focused repo pass for correctness, contradictory documentation, and high-leverage feature opportunities. Build/lint/check all pass on this branch.

Error-level content/data issues
  • Placeholder profile data in production settings: site/src/content/settings/main.json still contains a generic portfolio voice ("Creative professional..."), placeholder email hello@example.com, and an empty bookingUrl. This conflicts with the documented mission and blocks frictionless booking.
Confusing/contradictory documentation
  • Deployment contradiction: docs/architecture.md documents GitHub Pages deployment, while docs/ai-workflows.md still describes a Vercel preview flow ("Vercel detects push → profesional-site.vercel.app rebuilds").
  • Booking contradiction: docs/demo-plan.md, docs/epic-briefs.md, and this audit previously referenced a PLACEHOLDER Calendly URL in code. Current code routes booking from bookingUrl in settings; the real issue is that the settings value is empty.
  • CMS decision still unresolved in docs: Pages CMS is the documented editor path in most docs, but Decap is still referenced as an active option in docs/epic-briefs.md, docs/demo-plan.md, and this audit. The Decap admin config was already removed from the repo — but the docs have not caught up. Contributors reading planning docs can still infer two "official" editing paths.
Contradictions + multiple sources of truth (explained)
Topic Source A Source B Why this is risky Single source of truth to choose
Deployment platform docs/architecture.md: GitHub Pages workflow docs/ai-workflows.md: Vercel preview rebuild narrative Contributors do not know where previews/live deploys are expected, so triage and release debugging become inconsistent. One deployment doc section that all other docs link to.
Booking setup status Planning docs/audit text: "PLACEHOLDER Calendly in code" Current code: booking is settings-driven; value is empty in settings JSON People may fix the wrong layer (code) instead of the actual missing data (content settings). Settings schema + settings file should be the truth for booking readiness.
Editor/CMS path Pages CMS docs describe this as the active collaborator flow Planning docs still reference Decap as a live option despite its config having been removed from the repo Two valid-looking paths fragment onboarding and make automation/docs drift likely. A single "official editor path" decision record in docs + removal of Decap references from planning docs.

In practice, a multiple sources of truth problem means two places claim to be authoritative for one decision. For this repo, that creates avoidable execution churn (wrong fixes, wrong debugging path, and slower onboarding).

Useful features to prioritize next
  1. Single source of deployment truth: add a short "Hosting mode" section in README.md and link it from docs/architecture.md + docs/ai-workflows.md.
  2. Settings validation guard: add a CI check that fails when placeholder email or empty required booking/contact fields remain in production config.
  3. PR preview links: add preview deploy comments on pull requests to tighten review loops for content and design changes.
  4. Editorial quality gates: add markdown/content linting for docs + content folders with mission-language checks to prevent regression to template copy.
